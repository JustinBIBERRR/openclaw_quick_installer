# Windows 极端环境基线风险清单

本文档是上线前测试的基线约束。目标不是描述理想状态，而是记录当前代码真实行为、已知阻断和最小可验证证据。

## 1. 安装链路真实依赖

当前便携版 `openclaw-installer.exe` 的核心依赖如下：

- Windows 10/11 x64
- PowerShell 可执行
- 管理员权限（安装链路需要）
- 网络可访问 npm / 镜像（至少安装阶段需要）
- Node.js `>=18`（若无则尝试自动安装）
- npm 全局安装能力（`npm install -g openclaw@latest`）
- OpenClaw CLI 可发现（`openclaw` 或 `openclaw.cmd`）

当前主链路并不硬依赖：

- Docker
- Git
- Bash
- Hyper-V

## 2. 已知上线阻断（P0）

以下问题在当前代码下会直接阻断“极端小白环境”一键安装目标：

1) `NoWinget` 场景阻断  
`install.ps1` 在 Node 缺失时仅尝试 `winget install OpenJS.NodeJS.LTS`，没有自动 MSI 降级链路。  
影响：Windows 商店组件损坏、策略禁用 `winget`、离线机型会直接失败。

2) 中文/空格安装路径主动判失败  
`commands.rs` 的 `check_path()` 当前将非 ASCII 和空格路径判为不合法。  
影响：中文用户名、中文目录或包含空格目录无法按“路径可选”通过。

3) npm 全局安装权限异常阻断  
`npm install -g openclaw@latest` 失败即中断安装。  
影响：受限用户、企业权限策略环境失败率高。

4) PATH 刷新时机与命令发现风险  
安装后通过 `PATH + %APPDATA%\npm + npm prefix` 多路径查找 `openclaw.cmd`。  
影响：新装后立即验证阶段可能出现“已装但未发现”的假失败。

## 3. 高风险边界（P1）

- `openclaw` 已安装但版本探测异常，导致重复安装或误判
- 端口 `18789` 被占用时启动体验和错误提示是否可恢复
- WebView2 探测异常时是否误报（当前实现存在“默认 true”回退）
- 网络受限导致 `npm`、模型验证、doctor 行为不稳定
- 配置损坏后的诊断、修复、重试链路完整性

## 4. 证据与验收要求

每个测试案例必须留存以下证据（自动或手动均可）：

- 运行 profile 名称、时间戳、执行机器标识
- 安装结果状态（成功/失败）
- 结构化错误码与 hint（若失败）
- 安装日志目录与关键日志文件
- `install-info.json` 或结果文件内容
- 端口与进程状态快照（启动链路相关）

若命中 P0 阻断，版本不得进入发布阶段，除非：

- 阻断项已修复并有回归用例；
- 或有明确发布豁免记录（风险 owner、影响范围、回滚预案）。

## 5. 回归优先级

- P0：每次版本必跑，失败即阻断上线
- P1：每次版本至少自动化全跑 + 虚机抽样
- P2：按版本节奏轮转抽样（至少每两版覆盖一次）
